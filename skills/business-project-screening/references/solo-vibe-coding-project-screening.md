# 单人 + Vibe Coding 项目筛选框架

## 一句话判断

单人 + Vibe Coding 最适合做：

> 窄人群、高痛点、低合规、低客服、可先人工交付、后续可工具化提效的项目。

它不适合一上来做平台、社区、通用 AI 工具、重合规 SaaS，或者还没验证付费就先完整开发的项目。

核心顺序：

```text
找到窄人群 -> 验证痛点 -> 人工收费 -> 沉淀方法 -> 工具提效 -> 产品化
```

原框架回答的是：

本文档额外回答的是：

- 是否适合用 Vibe Coding 做第一版。
- 是否应该先人工收费，而不是先做 SaaS、小程序或完整系统。
- 工具化是否只是提高交付效率，而不是项目本身唯一价值。

## 横纵象限

用两个轴先判断项目位置：

- 横轴：需求强度。问题是否真实、具体、频繁或高痛，且能找到 10 个明确目标用户。
- 纵轴：单人可交付性。项目是否低合规、低客服、可先人工交付，且后续能用工具提效。

```text
                         单人可交付性高
                              ↑
                              |
        B. 服务先行区          |       A. 最适合区
   有需求，但先别急着产品化     |   单人 + Vibe Coding 优先做
   先人工服务，再抽象工具       |
                              |
需求弱/不确定  ←---------------+--------------→  需求强/付费明确
                              |
        D. 自嗨区              |       C. 交付陷阱区
   好做但没人急着付钱           |   有需求但客服/合规/交付太重
   很容易变成 Demo 玩具         |
                              |
                              ↓
                         单人可交付性低
```

输出审查结果时，必须明确写出项目落在哪个区域：

```text
Quadrant（象限区域）: A / B / C / D
```

如果“当前产品形态”和“建议验证形态”不在同一个区域，要同时写出：

```text
当前应用形态：D 区
人工服务验证形态：B 区
```

不要只给结论，不给象限。否则用户无法看懂这套坐标模型是否被使用。

## 四个象限

### A 区：最适合区

特征：

- 需求明确，付费理由清楚。
- 一个人能交付，不需要先做完整系统。
- 可以先人工验证，再把流程模板化和工具化。

适合继续验证。例子：

- 小红书美甲美睫商家号笔记体检。
- 电商差评归因报告。
- B2B 销售会议纪要转跟进邮件。
- 咨询师销售私信复盘。
- 本地门店差评修复方案。

### B 区：服务先行区

特征：

- 需求可能存在，但付费意愿或交付标准还不稳定。
- 用户愿意聊、愿意试，但未必愿意现在付钱。

处理方式：

- 不要先做 SaaS。
- 先人工服务 10-30 个用户。
- 把交付过程沉淀成清单、模板、评分规则或案例库。
- 再决定是否自动化。

### C 区：交付陷阱区

特征：

- 用户确实痛，但交付会变重。
- 客服、定制、合规、售后会吞掉单人精力。

常见例子：

- 医疗健康建议。
- 金融投资建议。
- 法律咨询。
- 社交产品。
- 复杂企业 SaaS。
- 需要大量人工审核的平台。

默认谨慎，除非能明显降低交付和合规负担。

### D 区：自嗨区

特征：

- 容易做 Demo。
- 用户觉得“有意思”，但没有强付费理由。
- 没有明确使用场景，也没有可触达用户。

常见例子：

- 泛 AI 写作工具。
- 通用效率工具。
- 泛内容生成器。
- “给所有人用”的 AI 小工具。
- 只有评分、没有结果交付的工具。

通常不建议继续投入。

## 7 条筛选标准

一个项目越符合下面 7 条，越适合单人 + Vibe Coding：

```text
1. 极窄人群
2. 高频或高痛
3. 低合规
4. 低客服
5. 可先人工交付
6. 工具只是提高交付效率
7. 第一版能找到 10 个明确付费用户
```

### 1. 极窄人群

不要说“小红书用户”，要说“本地美甲美睫门店老板”。

不要说“电商卖家”，要说“Etsy 手工饰品卖家”。

人群越窄，需求、话术、案例、渠道、价格越容易成立。

### 2. 高频或高痛

好项目至少满足一个：

- 高频：用户经常遇到。
- 高痛：不解决会带来明显损失。

最好两个都有。例如：小红书笔记不带来预约，门店老板会损失客源。

### 3. 低合规

早期尽量避开：

- 医疗。
- 金融。
- 法律。
- 教育资质。
- 未成年人。
- 敏感个人数据。
- 支付清算。
- 强监管行业。

### 4. 低客服

低客服项目通常是：

- 报告型。
- 模板型。
- 检查型。
- 改写型。
- 辅助决策型。

高客服项目通常是：

- 承诺结果。
- 需要持续陪跑。
- 需要大量定制。
- 用户不会自己操作。
- 出错后损失很大。

### 5. 可先人工交付

如果一个项目必须先做完整 SaaS 才能验证，风险很高。

更好的路径是：

```text
人工交付 -> 半自动交付 -> 工具化 -> 产品化
```

先卖结果，不要先卖系统。

### 6. 工具只是提高交付效率

AI 和代码不是价值本身。价值应该是：

- 帮用户多成交。
- 帮用户少返工。
- 帮用户省时间。
- 帮用户少踩坑。
- 帮用户提升内容、销售或交付质量。

工具只是让交付更快、更稳定。

### 7. 第一版能找到 10 个明确付费用户

不要先问市场大不大。先问：

> 我能不能在 7 天内找到 10 个目标用户，并让其中 1-3 个付费？

如果不能，先别做产品。

## 快速评分表

每项 0-10 分，总分 70 分。

| 维度 | 问题 | 得分 |
|---|---|---|
| 人群是否足够窄 | 能不能一句话说清楚目标用户？ | /10 |
| 痛点是否明确 | 用户不解决是否有损失？ | /10 |
| 付费理由是否清楚 | 用户为什么现在付钱？ | /10 |
| 是否低合规 | 是否避开高监管领域？ | /10 |
| 是否低客服 | 交付后是否可控？ | /10 |
| 是否可人工交付 | 不做系统能不能先卖？ | /10 |
| 是否能工具化 | 交付过程能不能模板化、自动化？ | /10 |

判断：

```text
60-70 分：优先验证
45-59 分：可以继续，但必须收窄
30-44 分：暂停，先重定义人群和付费场景
0-29 分：不建议做
```

## 案例：小红书美甲美睫笔记体检

项目定义：

> 面向本地美甲美睫门店，诊断小红书笔记为什么没有带来私信、预约和到店，并提供可直接发布的改写方案。

### 7 条标准判断

| 标准 | 是否符合 | 说明 |
|---|---|---|
| 极窄人群 | 符合 | 本地美甲美睫门店商家号 |
| 高频或高痛 | 基本符合 | 门店持续发内容，笔记不转化会影响客源 |
| 低合规 | 符合 | 只做内容诊断，不碰医疗金融法律 |
| 低客服 | 中等符合 | 报告型交付可控，但不能承诺结果 |
| 可先人工交付 | 符合 | 可以先人工诊断 10-30 篇笔记 |
| 工具只是提效 | 符合 | SaaS/小程序只是自动化诊断流程 |
| 能找到 10 个付费用户 | 待验证 | 这是当前最大未知数 |

初步结论：

```text
项目匹配度：75/100
付费验证前景：45-55%
结论：可以继续，但不要先做完整 SaaS
```

正确卖法：

不要卖：

> AI 小红书评分工具

要卖：

> 我帮你找出这篇笔记为什么不带预约，并给你 3 个可以直接发布的改写版本。

推荐漏斗：

```text
用户提交笔记链接/截图/标题正文
↓
免费获得 5 维成交力评分
↓
付费获得 3 次改写方案
↓
高意向用户转人工 1 对 1 诊断
```

5 个评分维度：

```text
1. 选题成交力
2. 标题点击力
3. 封面停留力
4. 正文信任力
5. 预约引导力
```

## 停止条件

出现下面情况，就不要继续投入开发：

- 30 个目标用户里没人愿意付费。
- 用户只说“有意思”，但不买改写方案。
- 用户更关心拍摄、投流、达人探店，而不是笔记改写。
- 每单交付超过 45 分钟，且无法模板化。
- 你无法判断一篇笔记为什么不转化。
- 用户不认可你的诊断结论。

## 最终原则

单人 + Vibe Coding 的正确顺序不是：

```text
想法 -> 做 SaaS -> 找用户 -> 想办法变现
```

而是：

```text
找到窄人群 -> 验证痛点 -> 人工收费 -> 沉淀方法 -> 工具提效 -> 产品化
```

Vibe Coding 降低的是开发门槛，不是商业门槛。真正值得做的项目，不是能不能做出来，而是能不能在不做完整产品前就有人愿意付钱。
