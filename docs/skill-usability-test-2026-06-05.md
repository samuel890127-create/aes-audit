# AES Audit Skill 可用性测试报告

测试日期：2026-06-05

## 测试结论

当前 skill 可用。

它能完成三个核心任务：

- 在用户写代码或做产品前，判断一个点子是否值得继续。
- 输出 `continue / pause / stop`，而不是泛泛鼓励。
- 对一人公司、vibe coding、AI SaaS、micro-SaaS 这类项目叠加单人执行约束，避免过早产品化。

当前最适合的使用方式：

```text
先用它杀掉明显不该做的点子，再把剩下的点子缩小到 3-7 天最小付费验证。
```

## 场景测试

### 测试 1：宠物出行小程序

用户输入：

```text
$business-project-screening
我想做一个宠物出行小程序，帮养宠用户找顺风车、宠物友好司机和同城接送服务。
```

预期输出：

```text
Conclusion: stop 或 pause
```

原因：

- 两边市场有匹配问题，接近平台或本地供给项目。
- 信任、安全、责任、客服和纠纷处理都很重。
- 一人公司很难在 7 天内验证密度和履约。
- 即使能做出小程序，也不代表商业成立。

可用性判断：通过。这个场景能触发红灯模型和一人公司执行约束。

### 测试 2：小红书美甲美睫商家号笔记体检

用户输入：

```text
$business-project-screening
一人公司审查
我想做一个工具，让美甲美睫门店上传小红书笔记，系统自动给出成交力评分，并提供 3 个改写方案。
```

预期输出：

```text
Conclusion: continue 或 pause
```

原因：

- 人群足够窄。
- 门店内容不转化会影响预约。
- 可先人工诊断，不必先做完整 SaaS。
- 最大未知数是用户是否愿意为改写方案付费。

建议验证：

```text
先人工诊断 10-30 篇笔记，收 19.9-99 元验证改写方案付费，再考虑工具化。
```

可用性判断：通过。这个场景能触发新增的一人公司/vibe coding 参考文档。

### 测试 3：面向所有人的 AI 写作工具

用户输入：

```text
$business-project-screening
商业筛选
我想做一个 AI 写作 SaaS，所有人都可以用，输入主题就能生成小红书、公众号、短视频文案。
```

预期输出：

```text
Conclusion: stop 或 pause
```

原因：

- 用户太宽。
- 替代品便宜且成熟。
- 付费理由不清楚。
- “所有人都能用”会导致定位、获客、定价都失焦。

建议压缩：

```text
改成某个窄人群的具体付费结果，例如“本地美甲门店的小红书预约笔记改写”。
```

可用性判断：通过。skill 能识别泛 AI 工具的低效幻想。

### 测试 4：B2B 销售会议纪要转跟进邮件

用户输入：

```text
$business-project-screening
我想给小型 B2B 销售团队做一个服务，把客户会议记录整理成需求、风险、下一步和跟进邮件。
```

预期输出：

```text
Conclusion: continue
```

原因：

- 场景明确。
- 销售跟进直接影响收入。
- 可以先人工代整理，再沉淀模板和工具。
- 一人交付压力较低。

建议验证：

```text
找 10 个销售或创始人，按 99-299 元处理一次会议记录，看是否愿意复购。
```

可用性判断：通过。skill 能识别适合一人公司验证的服务先行项目。

## 发现的问题

### 问题 1：新手不知道该给什么信息

skill 能处理信息缺失，但新手如果只输入“我想做一个 AI 工具”，输出会偏保守。需要新手文档告诉用户至少提供：

- 目标用户。
- 使用场景。
- 用户现在怎么解决。
- 谁付钱。
- 想卖什么结果。
- 自己能怎么触达第一批用户。

处理：新增 `docs/beginner-usage-guide.md`。

### 问题 2：多轮测试时可能带入上一轮案例信息

如果用户连续测试多个类似项目，模型可能把上一轮出现过的信息误带入当前项目，例如当前输入没有提到某个后续变现方式，却在审查里提到它。

处理：在 `SKILL.md` 和 `templates.md` 中增加输入边界规则：

```text
只审查当前项目简述里出现的信息。
缺失信息写 unknown（未知）。
不要继承上一轮测试案例、目标用户、变现方式或风险假设。
```

### 问题 3：用户可能误解红灯为“一个红灯就停止”

原框架实际规则是：

```text
3 个以上红灯默认 stop
1-2 个红灯通常 pause，除非有强事实抵消
```

处理：新手文档中明确说明。

### 问题 4：用户可能误解 A 区为“可以马上写产品”

A 区代表值得进入最小付费验证，不代表马上做 SaaS。

处理：新手文档中明确强调：

```text
A 区 -> 3-7 天最小付费验证 -> 再决定是否产品化
```

## 总体判断

当前 skill 适合公开给初学者使用，但需要配套说明。否则用户容易把它当成“商业计划生成器”或“AI 产品推荐器”。

准确定位应该是：

```text
AES Audit 是一个项目筛选 skill，不是创业鼓励器。
它的作用是在 AI 写代码前，帮你判断这个点子是否值得进入一人公司的射程。
```

## 建议发布话术

```text
这个 skill 不是帮你把点子包装得更好听。
它是帮你判断：这个点子该继续、该缩小，还是该立刻停掉。

尤其适合想用 AI 写产品、做一人公司、做 micro-SaaS 的人。
先审查，再写代码。
```
