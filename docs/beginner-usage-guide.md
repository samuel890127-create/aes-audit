# AES Audit 新手使用文档

## 这个 Skill 是干什么的

AES Audit 是一个项目筛选 Skill。

它不是帮你写商业计划，也不是帮你把点子说得更好听。

它帮你判断：

```text
这个点子该继续、该缩小，还是该停止。
```

最适合用在这些场景：

- 你想做一人公司。
- 你想用 AI 写代码做产品。
- 你想做 micro-SaaS。
- 你有一个内容变现、工具、服务、AI 产品想法。
- 你怕自己做一半才发现方向错了。

## 使用前先记住一句话

```text
不要先写代码。
先判断这个项目值不值得进入 3-7 天最小付费验证。
```

## 怎么安装

把仓库里的 skill 复制到 Codex skills 目录：

```bash
mkdir -p ~/.codex/skills
cp -R skills/business-project-screening ~/.codex/skills/
```

如果 Codex 没有立刻识别，重新打开一个对话。

## 最简单的用法

直接这样问：

```text
/项目审查
我的项目是：
{把你的想法写在这里}
```

或者：

```text
/AESAudit
My project is:
{your idea}
```

## 你应该提供哪些信息

信息越具体，判断越准。

最好按这个格式写：

```text
/项目审查
我的项目是：

目标用户：

他们在什么场景下会用：

他们现在怎么解决这个问题：

谁会付钱：

我想卖给他们什么结果：

我怎么找到前 10 个用户：

我现在做到哪一步：
```

如果你还没有想清楚，也可以只写一句话。Skill 会保守推断，但结论会更粗。

## 输出怎么看

Skill 通常会输出：

- `continue`：可以继续，但通常只是进入小测试，不是马上做完整产品。
- `pause`：方向可能有机会，但现在太大、太重、太模糊，需要压缩。
- `stop`：不建议继续投入，至少不要按当前形态做。

不要把 `continue` 理解成“马上写代码”。

更准确的意思是：

```text
可以进入 3-7 天最小付费验证。
```

## 红黄绿灯模型

Skill 会先用红黄绿灯判断项目。

### 红灯

红灯代表强风险。

常见红灯：

- 没有明确损失，只是“有意思”。
- 付费人不清楚。
- 必须先做完整产品才能验证。
- 需要平台网络效应。
- 一个人 7 天内无法验证。

注意：

```text
不是一个红灯就必死。
3 个以上红灯，默认 stop。
1-2 个红灯，通常 pause。
```

### 黄灯

黄灯代表要缩小。

常见黄灯：

- 人群太宽。
- 付费理由不清楚。
- 获客方式不明确。
- 交付太重。
- 有竞品，但你说不清优势。

遇到黄灯时，不要硬做，要压缩：

```text
缩小人群
缩小场景
缩小交付
缩小价格
缩小验证周期
```

### 绿灯

绿灯代表可以测试。

常见绿灯：

- 用户已经在用笨办法解决。
- 问题会造成钱、时间、机会或转化损失。
- 你能找到 10-30 个目标用户。
- 可以设计一个付费动作。
- 交付能变成模板、清单、流程或工具。

## 一人公司 + Vibe Coding 模型

如果你的项目和一人公司、AI 写代码、AI SaaS、micro-SaaS 有关，Skill 会叠加第二个模型。

这个模型看两个轴：

```text
横轴：需求强度和付费明确度
纵轴：单人可交付性
```

最值得做的是 A 区：

```text
用户真的痛，也可能付费；
同时一个人能低成本、低客服、低合规地交付。
```

但 A 区也不是让你马上写 SaaS。

正确顺序是：

```text
进入 3-7 天最小付费验证
验证成立
再考虑 AI 写产品或工具化
```

## 示例 1：不适合直接做的项目

输入：

```text
/项目审查
我想做一个宠物出行小程序，帮养宠用户找顺风车、宠物友好司机和同城接送。
```

可能结论：

```text
stop 或 pause
```

原因：

- 这是平台型或本地履约项目。
- 信任、安全、纠纷、客服都很重。
- 一个人很难在 7 天内验证供需密度。
- 做出小程序不等于商业成立。

## 示例 2：更适合测试的项目

输入：

```text
/一人公司审查
我想给本地美甲美睫门店做小红书笔记体检，帮他们判断为什么笔记没有带来私信和预约，并提供 3 个改写方案。
```

可能结论：

```text
continue 或 pause
```

原因：

- 人群足够窄。
- 问题和预约转化有关。
- 可以先人工诊断，不需要先做 SaaS。
- 最大未知数是店主是否愿意为改写付费。

下一步不是写代码，而是：

```text
找 10-30 个店主，人工诊断笔记，测试 19.9-99 元改写方案是否有人买。
```

## 常见误区

### 误区 1：以为 continue 就是可以开干

不是。

`continue` 只代表可以进入小测试。

### 误区 2：只写“我想做一个 AI 工具”

太模糊。

至少要说清楚：

- 给谁用。
- 在什么场景用。
- 为什么现在愿意付钱。

### 误区 3：把点赞、收藏、评论当成付费验证

不算。

付费验证优先级：

```text
付钱 > 交定金 > 接受价格并约试用 > 提交真实资料 > 加联系方式 > 说感兴趣
```

### 误区 4：先做完整产品

最危险的顺序是：

```text
想法 -> 写代码 -> 做产品 -> 找用户 -> 想办法变现
```

更好的顺序是：

```text
想法 -> 项目审查 -> 付费验证 -> 人工交付 -> 沉淀流程 -> 工具化
```
